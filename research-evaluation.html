Evaluation

Modbus/TCP is used as the first baseline instantiation of the Liscere action-evaluation chain. This page reports an initial feasibility demonstration for one protocol and one process model. It shows that a decision can follow declared operational context, and not protocol validity alone.

The control pair

The pair holds the protocol operation, automation artefact, address, and value constant, and changes only the declared maintenance window. The same sensitive configuration write to the ALARM_HI_SP artefact returns two different decisions.

ALERT

Production

Same write, with maintenance_window set to false.

ALLOW

Maintenance

ALARM_HI_SP · OBS-R004

Negative controls

The maintenance window is not a blanket alert suppressor. An out-of-range setpoint write during maintenance still returns an alert (OBS-R001), and a normal in-range write during maintenance remains an allow (OBS-R000). All four constructed scenarios returned their expected decision and rule.

What it shows, and what it does not

The demonstration shows that the action-evaluation chain runs end to end with one declared operational-context dimension, and that constructed scenarios can be checked against explicit expected decisions and evidence records. It does not measure detection performance, false-positive behaviour, robustness, or generality across context dimensions.

The context is declared, not inferred. The baseline covers one protocol and one controlled process model, with policy rules and artefact mappings defined manually. Multi-protocol validation remains future work, and the laboratory carries no production enforcement.

The scenarios run in OT Lab, the controlled laboratory available on the OT Lab repository.
